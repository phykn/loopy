# Loopy Theory Cycle 20260520_114744

## Target

Define when unstable loop work can be promoted into a stable artifact.

## Claim

A Loopy loop can promote a result when the latest change survives verification.

## Evidence

- `THEORY_LOOPY.md` already says Loopy promotes only what survives.
- A theory revision should not become final just because it was written; it needs to survive objection.
- A build result should not become stable just because code changed; it needs tests, execution, or direct behavior evidence.
- A review result should not become stable just because feedback exists; it needs a concrete defect, accepted risk, or verified fix.

## Objection

"Survives verification" is necessary but not enough. A tiny wording improvement can survive verification without being worth promotion.

Promotion needs two gates: the result must survive its verification surface, and it must change the artifact's decision, boundary, or usable state.

## Revision

A Loopy result can be promoted only when it passes both gates:

- Survival gate: the change survives the verification surface for that loop type.
- Value gate: the change alters a decision, boundary, minimum condition, or usable state of the artifact.

If the change only improves wording, formatting, or confidence without changing use, keep it in the cycle note or stop without promotion.

## Why It Is Stronger

The revision prevents promotion from becoming a log-cleanup step.

It keeps final artifacts reserved for changes that alter how future work should decide, classify, act, or stop.

## Progress Toward Target

This strengthens the promotion rule.

Before: stable meant "survived verification."

After: stable means "survived verification and changed the artifact's usable decision surface."

This is stable enough to promote into `THEORY_LOOPY.md`.
