# Loopy Theory Cycle 20260520_115139

## Target

Align the Loopy main theory and the `loopy-theory` skill.

## Claim

`loopy-theory` should use the same shape as Loopy's core theory: unstable theory, loop, verification, stable theory.

## Evidence

- `THEORY_LOOPY.md` now says `Loopy = Loop(Unstable -> Verified -> Stable)`.
- `loopy-theory` still uses older operational terms like target, progress, and decision.
- Those terms are useful, but they hide the simpler main shape.
- A theory-making skill should make it obvious which writing is unstable, which check verifies it, and what becomes stable.

## Objection

If the skill removes all operational terms, it may become too vague to execute.

The skill needs operational detail, but the detail should be subordinated to the simple flow.

## Revision

Keep the operational steps, but rename the workflow around the core theory:

- Unstable: current claim, rough note, weak theory, or open question.
- Loop: claim, break, revise.
- Verification: evidence-based objection strong enough to reject the claim.
- Stable: the smallest surviving theory promoted into `THEORY_<name>.md`.

## Why It Is Stronger

This keeps the final theory simple while making the skill executable.

The theory stays beautiful; the skill carries the operational burden.

## Progress Toward Target

This changes the skill structure, not the main theory.

It is worth applying because the skill should embody the theory it is meant to use.

## Next Question

Should the skill require every cycle to name its unstable artifact and stable candidate explicitly?
