# Loopy Theory Cycle 20260520_114529

## Target

Rebuild Loopy's main theory around making theories, not forcing preferred conclusions.

## Claim

Loopy is a loop system for turning unstable thinking into stable theory.

## Evidence

- The current project started with a theory loop, not a build loop or task runner.
- The useful document split is already clear: cycle notes keep rough reasoning, while `THEORY_<name>.md` keeps only the surviving theory.
- The earlier wording "what the loop is trying to make true" failed because theory work should not force a desired conclusion.
- The strongest current behavior is promotion: a rough claim survives objection and becomes a cleaner final theory.

## Objection

"Turning unstable thinking into stable theory" is too narrow if Loopy later includes review, research, planning, or build skills.

It may describe `loopy-theory`, but not the larger Loopy family.

## Revision

Loopy is a skill system where each loop turns unstable work into a stable artifact by preserving state, testing the next change, and promoting only what survives.

For `loopy-theory`, the stable artifact is a theory.
For future skills, the stable artifact may be a review, plan, research conclusion, or implementation.

## Why It Is Stronger

The revision keeps theory-making as the first concrete case without trapping Loopy inside theory only.

It also avoids the false idea that a loop exists to make a preferred conclusion true. A Loopy loop exists to decide what can be promoted from rough work into stable output.

## Progress Toward Target

This changes the main boundary of Loopy.

Before: Loopy was mainly "goal-driven autonomous loops."

After: Loopy is "promotion through loops": unstable work becomes stable artifact only after state, verification, and decision.

This is strong enough to promote because it explains the current `loopy-theory` workflow and leaves room for future Loopy skills.
