# Loopy Theory Cycle 20260520_114956

## Target

Make Loopy's main theory simple and beautiful instead of procedural.

## Claim

Loopy turns unstable work into stable artifacts through loops.

## Evidence

- The current final theory contains useful rules, but it reads like an operating manual.
- The user wants the theory itself to be simple and beautiful.
- The strongest repeated structure is not the full rule list; it is the movement from rough work to stable artifact.
- Promotion, autonomy, and resumable state are important, but they are derived from that movement.

## Objection

The claim may become too simple. If it only says "unstable to stable," it may not distinguish Loopy from ordinary iteration.

The theory needs one more condition: stability must come from survival under verification, not from completion or preference.

## Revision

Loopy is a loop that lets unstable work become stable only by surviving verification.

Minimal form:

```text
Loopy = Loop(Unstable -> Verified -> Stable)
```

In words:

```text
Rough work stays in the loop.
Only what survives verification becomes stable.
```

## Why It Is Stronger

The revision is shorter and more general.

It still preserves the important boundary: Loopy is not any repetition, and it is not a checklist. It is promotion through verification.

## Progress Toward Target

This changes the final theory's shape.

Before: the theory listed operational rules.

After: the theory states the core movement and leaves detailed rules as consequences.

This should be promoted because it gives a simpler minimum claim.

## Next Question

Which details are true consequences of the core theory, and which are only implementation policy?
