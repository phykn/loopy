# Loopy Theory Cycle 20260520_131133

## Target

Verify whether Loopy should have a core philosophy that child skills inherit, and whether loop testing should use separate agent context only when independence matters.

## Claim

Loopy's core can be stated once as `Loop(Unstable -> Tested -> Stable)`, then specialized by child skills such as `loopy-theory` and future `loopy-review`.

## Evidence

- `loopy/CORE.md` defines the shared minimum: unstable artifact, change loop, rejection test, and stable survivor.
- `THEORY_LOOPY.md` already uses the same formula, so the core does not introduce a conflicting theory.
- `skills/loopy-theory/SKILL.md` now treats `loopy/CORE.md` as parent philosophy when present.
- The skill validator passed after the change.
- The user's concern was specifically about context separation during loops, not about splitting every task into agents.

## Objection

If every loop step uses a separate agent, Loopy becomes heavier without improving the rejection test. Agent separation is useful only when the critic's judgment must be insulated from the drafting context.

## Revision

Core inheritance should be mandatory, but agent isolation should be conditional:

```text
Use a separate agent context when the test depends on independent judgment.
Do not use a separate agent only to imitate independence.
```

## Revision target: theory or skill/process

Both.

## Why it is stronger

The revision preserves the simple core while preventing the process from becoming ritual. It lets theory, review, and future skills share one minimum condition without forcing every loop into the same operational shape.

## Decision

Promote.

## Process recommendation, if the loop process was weak but skill edits are not allowed

None. Skill edits were allowed because the current task is Loopy skill development.

## Progress toward target

`loopy/CORE.md` now defines the parent philosophy, and `loopy-theory` now inherits it while limiting agent separation to cases where independence improves testing.
